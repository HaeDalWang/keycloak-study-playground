Keycloak
과 IAM

발표자 배승도
01 · PROBLEM

우리가 겪는 문제

  • 직원 200명 × 사내 시스템 8개 = 계정 1,600개
  • 퇴사자 처리 누락 → 6개월 후에도 HR 시스템 접근 가능
  • 직원은 비밀번호 8개를 외워야 함 → 생산성 저하, 보안 사고
이걸 한 번에 해결하는 방법이 있습니다
AuthN/AuthZ → RBAC → SSO → OAuth2 → OIDC → Keycloak
02 · CONCEPT

IAM (Identity and Access Management)

  • "누가, 무엇에, 어떻게 접근할 수 있는지 관리하는 체계"
  • Identity: 당신이 누구인지 → 인증 (Authentication)
  • Access: 무엇을 할 수 있는지 → 인가 (Authorization)
  • AWS IAM을 이미 쓰고 있다면 — 그게 IAM이다

→ 새로운 문제: 사용자 수 증가 → 수천 개 권한을 일일이 설정하는 관리 과부하

03 · CONCEPT

인증 vs 인가

인증 (Authentication)

"당신이 누구인지 확인"

  • 로그인
  • 비밀번호, 지문, OTP
  • 신분증 확인

인가 (Authorization)

"무엇을 할 수 있는지 확인"

  • 권한 체크
  • 팀장만 결재 가능
  • 개발팀만 코드 저장소 접근
04 · CONCEPT

RBAC (Role-Based Access Control)

  • 나쁜 방식: 사람에게 직접 권한 부여 → 300명이면 300번 설정
  • 좋은 방식: 역할(Role)에 권한 → 역할을 사람에게 할당
  • 예시: ROLE_HR_ACCESS → 개발팀 그룹 → 김철수 자동 상속
  • 인사이동 시 그룹만 바꾸면 권한 자동 변경

→ 새로운 문제: 앱이 수십 개로 늘어나면서 앱마다 로그인 반복 — 로그인 피로도(Password Fatigue)

05 · CONCEPT

SSO (Single Sign-On)

  • 한 번 로그인 → 모든 시스템 접근
  • 퇴사 처리: 계정 1개 비활성화 → 전체 시스템 즉시 차단
  • 직원 생산성 향상, IT 운영 비용 절감
  • 비밀번호는 한 곳(IdP)에서만 관리
SSO = 건물 통행증
"나 이 건물(회사) 들어왔으니까 모든 방(앱) 다 열어줘"

→ 새로운 문제: 모바일 앱 확산 — 제3자 서비스에 권한을 '부분적으로' 위임해야 하는 상황

06 · ARCHITECTURE

IdP (Identity Provider)

  • 신원을 확인하고 토큰을 발급하는 서버
  • IdP가 토큰에 디지털 서명 → 서버는 IdP 공개키로 검증
  • 과거: 개발자가 앱 10개에 검증 로직을 직접 심어야 했음
  • Keycloak: "내가 대신 검증해줄게" — 검증기를 한 곳에 집중

→ 그렇다면 IdP가 발행하는 토큰, 실제로 어떻게 생겼나?

07 · PROTOCOL

SAML 2.0 — IdP가 쓰던 프로토콜

  • 2005년 기업 표준 — IdP와 앱이 대화하는 방식
  • XML 기반 티켓(Assertion) — 브라우저 POST로만 전달
  • 아직도 현역: SAP, Oracle, 정부 시스템, 오래된 그룹웨어
  • 문제: 모바일 앱·API 환경에서 브라우저 없으면 작동 불가

→ 새로운 문제: XML이 너무 무겁고 브라우저 전용 — 모바일/API 시대에 맞지 않음

08 · TOKEN FORMAT

토큰 형태의 진화

  • SAML (옛날): XML 덩어리 — 태그가 절반, 브라우저 POST로만 전달
  • JWT (현대): 점 3개로 구분된 문자열 — HTTP 헤더에 쏙 들어감
  • JWT는 자가 수용적(Self-contained) — 토큰 자체가 정보를 들고 있음
  • 서버가 DB 안 뒤져도 됨 — 서명만 검증하면 끝 (Stateless)
형태 변화 = 모바일/클라우드 시대로의 전환
브라우저 전용 → 모든 기기 공용

→ 새로운 문제: 모바일 앱 확산 — 제3자 서비스에 권한을 '부분적으로' 위임해야 하는 상황

09 · PROTOCOL

OAuth2 — 위임 인증

  • 앱이 사용자 비밀번호를 직접 받지 않고 인증을 IdP에 위임하는 표준
  • SSO와 결정적 차이: SSO는 "내가 로그인", OAuth2는 "앱이 나 대신 일함"
  • Scope: 권한을 쪼갤 수 있음 — "읽기만", "특정 폴더만"
  • Refresh Token: 유저 없이도 앱이 배경에서 토큰 갱신 가능
OAuth2 = 청소업체 카드키
"청소업체(서드파티 앱)한테 거실만 청소할 수 있는 일회용 카드키를 줄게"

→ 새로운 문제: OAuth2는 인가 프레임워크 — "이 사람이 누구인지" 표준화된 신원 정보가 없다

10 · SECURITY

Refresh Token — 보안 딜레마 해결

  • 문제 A: 토큰 유효기간 짧게 → 10분마다 재로그인 → 서비스 망함
  • 문제 B: 유효기간 1년 → 탈취 시 1년간 해커가 주인
  • OAuth2 해결책: 토큰을 두 종류로 분리
  • Access Token(5분) + Refresh Token(30일) — 역할 분담
RT Rotation (최신 트렌드)
RT 한 번 쓰면 새걸로 교체 → 탈취된 RT 재사용 시 즉시 계정 잠금

→ 새로운 문제: OAuth2는 인가 프레임워크 — "이 사람이 누구인지" 표준화된 신원 정보가 없다

11 · PROTOCOL

OIDC (OpenID Connect)

  • OAuth2 위에 신원 정보를 추가한 확장 표준
  • OAuth2: scope(권한)만 — 사용자가 누군지 모름
  • OIDC: ID Token 추가 — 이름·이메일·역할 포함
  • 실무에서 Keycloak 연동 = 거의 항상 OIDC
현재 표준 — 클라우드·분산 시스템의 사실상 표준(De facto)
다음 방향: Passwordless · Passkey로 보안 강화 중
12 · SOLUTION

Keycloak

Red Hat이 만든 오픈소스 엔터프라이즈 IdP

  • SSO · LDAP/AD · 소셜 로그인 · MFA 올인원
  • OIDC + SAML 2.0 동시 지원
  • 폐쇄망 완전 운영 — 인터넷 불필요
  • 로그인 화면 완전 커스텀 (화이트라벨)
13 · ARCHITECTURE

Keycloak 논리 구조

Realm격리된 인증 도메인 — 회사 전체 (예: ezl)
Client보호 대상 앱 — HR, 포털, 결재
Role권한 단위 — ROLE_HR_ACCESS
Group사용자 묶음 — 개발팀, 인사팀
User실제 사람 — 그룹 소속 → Role 자동 상속
