발표자 배승도
01 · PROBLEM

우리가 겪는 문제

  • 직원 200명 × 사내 시스템 8개 = 계정 1,600개
  • 퇴사자 처리 누락 → 6개월 후에도 HR 시스템 접근 가능
  • 직원은 비밀번호 8개를 외워야 함 → 생산성 저하, 보안 사고
  • IT 헬프데스크 문의의 40%가 비밀번호 재설정
이걸 한 번에 해결하는 방법이 있습니다
02 · CONCEPT

IAM (Identity and Access Management)

  • "누가, 무엇에, 어떻게 접근할 수 있는지 관리하는 체계"
  • Identity: 당신이 누구인지 → 인증 (Authentication)
  • Access: 무엇을 할 수 있는지 → 인가 (Authorization)
  • AWS IAM을 이미 쓰고 있다면 — 그게 IAM이다

→ 인증과 인가, 어떻게 다른가?

03 · CONCEPT

인증 vs 인가

인증 (Authentication)

"당신이 누구인지 확인"

  • 로그인
  • 비밀번호, 지문, OTP
  • 신분증 확인

인가 (Authorization)

"무엇을 할 수 있는지 확인"

  • 권한 체크
  • 팀장만 결재 가능
  • 개발팀만 코드 저장소 접근
04 · CONCEPT

RBAC (Role-Based Access Control)

  • 나쁜 방식: 사람에게 직접 권한 부여 → 300명이면 300번 설정
  • 좋은 방식: 역할(Role)에 권한 → 역할을 사람에게 할당
  • 예시: ROLE_HR_ACCESS → 개발팀 그룹 → 김철수 자동 상속
  • 인사이동 시 그룹만 바꾸면 권한 자동 변경

→ 이걸 여러 시스템에 한 번에 적용하려면?

05 · CONCEPT

SSO (Single Sign-On)

  • 한 번 로그인 → 모든 시스템 접근
  • 퇴사 처리: 계정 1개 비활성화 → 전체 시스템 즉시 차단
  • 직원 생산성 향상, IT 운영 비용 절감
  • 비밀번호는 한 곳에서만 관리

→ 그러면 이 SSO를 누가 처리해주는가?

06 · ARCHITECTURE

IdP (Identity Provider)

  • 신원을 확인하고 토큰을 발급하는 서버
  • 비유: 호텔 프런트(IdP)가 카드키(토큰) 발급 → 레스토랑/수영장(앱)은 카드키만 확인. 비밀번호는 프런트만 앎
  • 외부 IdP: Google, Kakao — 이미 쓰고 있는 개념
  • 기업 자체 IdP 필요 이유: 폐쇄망, 사내 AD 연동, 외부 의존 제거

→ 그러면 IdP와 앱은 어떤 언어로 대화하는가?

07 · PROTOCOL

OAuth2 — 위임 인증

  • 앱이 사용자 비밀번호를 직접 받지 않고 인증을 IdP에 위임하는 표준
  • 비밀번호는 IdP만 알고, 앱은 토큰만 받음
  • 토큰 = 카드키. 만료 시간 있음 (보통 5분)
  • 신규 앱 개발의 사실상 표준

→ 그런데 OAuth2만으로는 "이 사람이 누구인지"를 모른다

08 · PROTOCOL

OIDC (OpenID Connect)

  • OAuth2 위에 신원 정보를 추가한 확장 표준
  • OAuth2: "이 사람 접근 허용" (권한)
  • OIDC: "이 사람은 김철수, 팀장" (신원 + 권한)
  • JWT 토큰에 사용자 정보 포함 — 앱이 별도 조회 불필요
  • 실무에서 Keycloak 연동 = 거의 항상 OIDC
09 · PROTOCOL

SAML 2.0 — 레거시 기업 표준

  • XML 기반, 2005년부터 기업 표준
  • 아직도 많이 씀: SAP, Oracle, 정부 시스템, 오래된 그룹웨어
  • OIDC vs SAML: 신규 개발은 OIDC, 레거시 연동은 SAML
  • Keycloak은 OIDC와 SAML 2.0을 동시에 지원
10 · SOLUTION

Keycloak

Red Hat이 만든 오픈소스 엔터프라이즈 IdP

  • SSO · LDAP/AD · 소셜 로그인 · MFA 올인원
  • OIDC + SAML 2.0 동시 지원
  • 폐쇄망 완전 운영 — 인터넷 불필요
  • 로그인 화면 완전 커스텀 (화이트라벨)
