EZL 통합 인증(SSO) 시스템 구축 요구사항 체크리스트
[전제 조건 확인 완료] HA 이중화 구성 / 폐쇄망 환경 / AD 연동

A. 서버 및 인프라 환경
Keycloak 서버 구성 수량 (HA 구성 시 최소 2대)
확인 목적: 장애 발생 시 서비스 중단을 방지하는 이중화 아키텍처를 구성하고, 서버 간 실시간 데이터(세션) 동기화 방식을 확정하기 위함입니다.

각 서버 상세 스펙 (CPU, RAM, 디스크)
확인 목적: 준비된 인프라 자원에 맞춰 통합 인증 시스템이 최상의 성능을 낼 수 있도록 메모리 할당량을 최적화하기 위함입니다.

운영 체제(OS) 종류 (RHEL / Rocky / Ubuntu 등)
확인 목적: 서버 재부팅 시에도 인증 서비스가 자동으로 복구 및 실행되도록 OS 환경에 맞는 구동 스크립트를 적용하기 위함입니다.

데이터베이스(DB) 종류 (PostgreSQL 권장) 및 구성 위치
확인 목적: 사용자의 중요 인증 데이터와 권한 정보를 안전하게 영구 저장할 DB를 구성하고, 방화벽 및 접근 통제 정책을 수립하기 위함입니다.

서버 간 내부 통신 IP 대역
확인 목적: 다중 서버 환경에서 사용자의 로그인 상태(세션)가 끊기지 않도록, 서버 간에만 통신하는 안전한 전용 동기화 네트워크를 설정하기 위함입니다.

B. 도메인 및 접속 환경
Keycloak 서비스 접속 도메인 (예: auth.company.co.kr)
확인 목적: 임직원들이 통합 로그인을 위해 접속할 공식 주소를 시스템에 등록하여, 위조된 사이트(피싱) 접속을 원천 차단하기 위함입니다.

전면 로드밸런서(L4/L7) 또는 웹 서버(Nginx 등) 유무
확인 목적: 접속 트래픽이 로드밸런서를 거치더라도 시스템이 실제 사용자의 IP와 네트워크 상태를 정확하게 인지하고 감사 로그를 남기기 위함입니다.

SSL 인증서 발급 방식 (내부 CA 또는 공인 인증서)
확인 목적: 사용자가 로그인할 때 입력하는 계정 정보가 네트워크상에서 암호화(HTTPS)되어 안전하게 전송되도록 보안 통신을 구성하기 위함입니다.

내부 사용자 접속 포트 (기본 443 또는 8443)
확인 목적: 사내 네트워크 보안 정책 및 방화벽 설정에 위배되지 않는 안전하고 표준화된 웹 접속 경로를 개방하기 위함입니다.

C. 사내 AD/LDAP 연동
AD 서버 IP 및 LDAPS(암호화 통신) 지원 여부
확인 목적: 시스템이 사내 인사 정보를 읽어올 경로를 지정하며, 이때 임직원의 정보가 암호화된 상태로 안전하게 전송될 수 있는지 확인하기 위함입니다.

Keycloak 전용 AD 서비스 계정(읽기 전용) 발급 가능 여부
확인 목적: 통합 인증 시스템이 임의로 사내 인사 정보를 수정하지 못하도록, 권한이 통제된 조회 전용 시스템 계정으로만 접근하게 하여 보안을 강화하기 위함입니다.

사용자/부서 그룹이 위치한 AD 경로(OU) 및 총 사용자 수
확인 목적: 전체 AD 조직도 중 실제 SSO가 필요한 부서와 직원의 데이터만을 선별하여 연동하고, 데이터 규모에 맞춘 최적의 동기화 주기를 설정하기 위함입니다.

임직원 로그인 ID 형식 (사번, 이메일 등) 및 퇴사자 처리 정책
확인 목적: 사용자가 익숙하게 사용하는 ID 형식으로 로그인을 지원하고, 퇴사자 발생 시 연동된 모든 시스템의 접근 권한을 즉시 차단하기 위함입니다.

D. 연동할 애플리케이션 (SSO 적용 범위)
SSO를 적용할 사내 시스템 목록 및 각 접속 URL
확인 목적: 한 번의 로그인으로 접근할 수 있는 대상 시스템을 확정하고, 로그인 성공 후 정상적인 목적지로만 이동(Redirect)하도록 보안 허용 목록을 등록하기 위함입니다.

각 시스템의 로그인 처리 방식 (웹, API 호출, 레거시 환경 등)
확인 목적: 대상 시스템의 구조에 맞춰 가장 안전하고 적합한 국제 표준 인증 프로토콜(OIDC, SAML 등)을 적용하기 위함입니다.

대상 시스템의 개발/운영 담당자 유무
확인 목적: SSO 연동 작업 시 대상 시스템 측에 제공해야 할 암호화 키(Secret) 및 연동 개발 가이드를 사전에 준비하여 원활한 협업을 진행하기 위함입니다.

E. 조직 및 권한 구조
시스템별 접근 권한 차등 적용 여부 (부서별/직급별 권한)
확인 목적: '인사팀만 HR 시스템 접근', '개발팀만 Jira 접근'과 같이 부서와 직무 역할(Role)에 따른 엄격한 접근 통제(Access Control) 체계를 설계하기 위함입니다.

통합 인증 시스템(Keycloak) 관리자 배정
확인 목적: 구축 완료 후 전체 시스템의 보안 정책과 계정을 관리할 최고 관리자 권한을 안전하게 인계하기 위함입니다.

F. 비밀번호 및 보안 정책
비밀번호 복잡도 (길이, 대소문자/특수문자 조합) 및 유효 기간
확인 목적: 사내 정보보안 감사 규정에 부합하는 수준으로 임직원의 비밀번호 생성 및 주기적 변경을 강제하기 위함입니다.

이전 비밀번호 재사용 금지 및 최초 로그인 시 변경 강제 여부
확인 목적: 유출되었을 가능성이 있는 과거의 비밀번호를 다시 사용하지 못하게 차단하고, 초기 발급된 임시 비밀번호의 보안 취약점을 제거하기 위함입니다.

G. 세션 및 로그인 정책
유휴 자동 로그아웃 시간 및 최대 로그인 유지 시간
확인 목적: 자리를 비우거나 장시간 미사용 시 자동으로 세션을 종료하여, 타인에 의한 단말기 및 사내 시스템 무단 도용을 방지하기 위함입니다.

로그인 실패 횟수 제한 및 계정 잠금 정책
확인 목적: 해커나 악성 봇이 비밀번호를 무작위로 대입하여 계정을 탈취하는 공격(Brute Force)을 원천 차단하기 위함입니다.

H. MFA (2차 인증)
OTP(2차 인증) 도입 여부 및 적용 대상 그룹
확인 목적: 사외망 접속자나 주요 권한 보유자(관리자, 재무팀 등)를 대상으로 추가적인 인증 절차를 요구하여 핵심 시스템의 보안 등급을 상향하기 위함입니다.

사용 예정인 OTP 애플리케이션 (Google, MS Authenticator 등)
확인 목적: 임직원들의 모바일 기기에 설치할 OTP 앱과 통합 인증 시스템 간의 호환성을 검증하기 위함입니다.

I. 토큰 및 커스터마이징
대상 시스템에서 요구하는 사용자 추가 정보 (사번, 부서, 직급 등)
확인 목적: 연동된 시스템들이 사용자를 정확히 식별하고 내부 권한을 부여할 수 있도록, 로그인 완료 시 전달되는 인증 토큰(Token)에 필요한 인사 데이터를 포함시키기 위함입니다.

J. 운영 및 알림
감사 로그 저장 위치 (로컬, 중앙 로그 서버, SIEM)
확인 목적: 보안 사고 발생 시 추적을 위한 사용자 로그인 이력 및 시스템 관리 기록(Audit Log)을 안전한 곳에 영구 보존하기 위함입니다.

장애 발생 시 알림 매체 및 정기 백업 정책
확인 목적: 시스템 오픈 이후 안정적인 서비스 운영을 위해 장애 조기 탐지 체계를 구축하고, 재난 발생 시 신속한 데이터 복구 방안을 마련하기 위함입니다.
