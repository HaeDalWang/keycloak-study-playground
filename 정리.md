
### 1. 초기 단계: Authentication(인증)과 Authorization(인가)의 분리

- **배경:** 초창기 컴퓨팅 환경에서는 각 서버나 어플리케이션이 개별적으로 사용자 정보를 파일이나 자체 DB에 저장했습니다.
    
- **기존 문제점:** 인증(누구인가?)과 인가(무엇을 할 수 있는가?)의 개념이 혼재되어 보안 정책을 일관되게 적용하기 어려웠습니다.
    
- **해결 매커니즘:**
    
    1. **Authentication (AuthN):** ID/PW 등을 통해 사용자의 신원을 확인하는 절차를 정립했습니다.
        
    2. **Authorization (AuthZ):** 신원이 확인된 사용자에게 특정 자원에 대한 접근 권한을 부여하는 절차를 논리적으로 분리했습니다.
        
- **새로운 문제:** 사용자 수가 늘어남에 따라 각 사용자별로 수천 개의 자원에 대한 권한을 일일이 설정하는 '관리의 과부하'가 발생했습니다.

### 2. RBAC (Role-Based Access Control, 1990년대 표준화)

- **배경:** 수천 명의 직원에게 개별적으로 권한을 부여하는 방식은 실수가 잦고 퇴사/부서 이동 시 권한 회수가 누락되는 보안 허점이 컸습니다.
    
- **해결 매커니즘 (Step-by-Step):**
    
    1. **추상화:** 권한(Permission)을 사용자에게 직접 붙이지 않고 '역할(Role)'이라는 중간 객체에 묶습니다.
        
    2. **할당:** 사용자를 해당 역할에 할당합니다 (예: '개발자' 역할 부여).
        
    3. **상속/매핑:** 사용자는 자신에게 부여된 역할을 통해 그 역할이 가진 모든 권한을 수행합니다.
        
- **새로운 문제:** 기업 내 어플리케이션이 수십 개로 늘어나면서, 사용자는 앱마다 로그인해야 하는 '로그인 피로도(Password Fatigue)'와 파편화된 계정 관리 문제가 제기되었습니다. 이를 해결하기 위해 **IAM(Identity & Access Management)** 개념이 통합 관리 체계로 부상했습니다.

### 3. SSO (Single Sign-On) 및 IdP (Identity Provider)

- **배경:** 여러 시스템을 하나의 계정으로 로그인하려는 니즈가 커졌습니다.
    
- **해결 매커니즘 (SAML 중심, 2000년대 초반):**
    
    1. **신뢰 관계 구축:** 서비스 제공자(SP)와 신원 제공자(IdP) 간에 인증 정보를 주고받을 신뢰 관계를 설정합니다.
        
    2. **인증 위임:** 사용자가 SP에 접속하면, SP는 사용자를 IdP(중앙 인증 서버)로 리다이렉트합니다.
        
    3. **토큰 발행:** IdP가 인증을 완료하면 인증 결과가 담긴 토큰(SAML Assertion 등)을 발행하고, SP는 이를 검증하여 세션을 생성합니다.
        
- **새로운 문제:** SSO는 웹 브라우저 기반의 기업 내부망 중심이었습니다. 모바일 앱 확산과 제3자 서비스(API)에 내 데이터 접근 권한을 '부분적으로' 빌려줘야 하는 상황(Delegation)에는 대응하기 어려웠습니다.

### 4. OAuth 2.0 (RFC 6749, 2012년)

- **배경:** 외부 앱(Third-party)이 사용자의 ID/PW를 직접 알지 못하게 하면서도, 사용자의 자원(예: 구글 캘린더, 사진)에 접근할 수 있게 하는 '권한 위임'이 필요해졌습니다.
    
- **해결 매커니즘 (Step-by-Step):**
    
    1. **Authorization Request:** 클라이언트 앱이 자원 소유자(사용자)에게 권한 승인을 요청합니다.
        
    2. **Authorization Grant:** 사용자가 승인하면 클라이언트는 '인가 코드'를 받습니다.
        
    3. **Access Token 교환:** 클라이언트는 이 코드를 Authorization Server에 제출하고 **Access Token**을 발급받습니다.
        
    4. **자원 접근:** 클라이언트는 ID/PW 대신 이 토큰만 API 서버에 제시하여 데이터를 가져옵니다.
        
- **새로운 문제:** OAuth 2.0은 **인가(Authorization)** 프레임워크일 뿐, **인증(Authentication)** 프로토콜이 아닙니다. 즉, 토큰을 가진 자가 '누구인지'에 대한 표준화된 신원 정보(이름, 이메일 등)를 제공하지 않아 개발자들이 제각각 구현하는 혼선이 생겼습니다.

### 5. OIDC (OpenID Connect, 2014년)

- **배경:** OAuth 2.0의 보안성을 유지하면서, 사용자의 신원 정보(Profile)를 표준화된 방식으로 가져올 필요가 있었습니다.
    
- **해결 매커니즘:**
    
    1. **ID Token 도입:** OAuth 2.0의 Access Token 외에 ID Token (JWT 형태)을 추가로 발행합니다.
        
    2. **표준 클레임:** ID Token 내부에는 사용자의 식별자(sub), 이름, 이메일 등 표준화된 필드가 포함되어 있어 별도의 API 호출 없이도 신원 확인이 가능합니다.
        
    3. **UserInfo Endpoint:** 추가 정보가 필요한 경우 표준화된 경로로 신원 정보를 요청할 수 있게 규격화했습니다.
        
- **현재 상태:** 현대의 분산 시스템과 클라우드 환경에서는 OIDC가 사실상의 표준(De facto standard)으로 자리 잡았으며, 최근에는 무자격 증명(Passwordless), Passkey 등을 통해 보안을 더욱 강화하는 방향으로 발전 중입니다.
