# 챕터 10: SAML 2.0 연동

> 가상 시나리오: 레거시 그룹웨어가 SAML 2.0만 지원 → Keycloak을 SAML IdP로 연결

---

## 1. SAML vs OIDC

```
OIDC:  JSON + JWT. 가볍고 현대적. 신규 개발 표준
SAML:  XML 기반. 무겁지만 기업 레거시 시스템 표준 (2005년~)
공통점: 둘 다 SSO 구현 가능. Keycloak이 둘 다 지원.
```

**SAML SP-Initiated 흐름**:
```
1. 사용자 → 그룹웨어(SP) 접속
2. 그룹웨어 → Keycloak(IdP)으로 SAMLRequest 전송
3. 사용자 → Keycloak 로그인
4. Keycloak → 그룹웨어로 SAMLResponse(Assertion) 전송
5. 그룹웨어 → Assertion 검증 → 접근 허용
```

---

## 2. 메타데이터 교환 (실무 프로세스)

```
[1단계] Keycloak IdP 메타데이터 → 레거시 벤더에게 전달
  URL: http://<keycloak>/realms/jtbc/protocol/saml/descriptor

[2단계] 레거시 벤더 SP 메타데이터 → Keycloak에 SAML Client로 등록
  벤더가 주는 것: Entity ID + ACS URL (+ 서명 인증서)
```

### IdP 메타데이터 핵심 항목

| 항목 | 의미 | OIDC 대응 |
|------|------|---------|
| `entityID` | Keycloak 식별자 | `issuer` |
| `SSO URL` | 로그인 엔드포인트 | `authorization_endpoint` |
| `SLO URL` | 로그아웃 엔드포인트 | `end_session_endpoint` |
| `X509Certificate` | 서명 검증용 공개키 | JWKS 공개키 |

### Keycloak에 SAML Client 등록

```bash
# REST API로 SAML Client 생성
curl -X POST http://keycloak/admin/realms/jtbc/clients \
  -d '{
    "clientId": "http://legacy-groupware.jtbc.co.kr/saml",  # SP Entity ID
    "protocol": "saml",
    "redirectUris": ["http://legacy-groupware.jtbc.co.kr/saml/acs"],  # ACS URL
    "attributes": {
      "saml.server.signature": "true",
      "saml.assertion.signature": "true",
      "saml_name_id_format": "username"
    }
  }'
```

## 3. SAML Assertion 구조 (JWT의 XML 버전)

```xml
<saml:Assertion>

  <!-- JWT의 iss에 해당 -->
  <saml:Issuer>http://localhost:8080/realms/jtbc</saml:Issuer>

  <!-- JWT의 Signature에 해당 -->
  <ds:Signature>
    <ds:SignatureValue>RSA 서명값 (Base64)...</ds:SignatureValue>
    <ds:X509Certificate>공개키 인증서...</ds:X509Certificate>
  </ds:Signature>

  <!-- JWT의 sub, exp에 해당 -->
  <saml:Subject>
    <saml:NameID>kim.cheolsu</saml:NameID>
    <saml:SubjectConfirmationData
      NotOnOrAfter="2026-05-03T10:00:00Z"
      Recipient="http://legacy-groupware.jtbc.co.kr/saml/acs"/>
  </saml:Subject>

  <!-- JWT의 iat, exp, aud에 해당 -->
  <saml:Conditions
    NotBefore="2026-05-03T09:55:00Z"
    NotOnOrAfter="2026-05-03T10:00:00Z">
    <saml:AudienceRestriction>
      <saml:Audience>http://legacy-groupware.jtbc.co.kr/saml</saml:Audience>
    </saml:AudienceRestriction>
  </saml:Conditions>

  <!-- JWT의 Claim에 해당 -->
  <saml:AttributeStatement>
    <saml:Attribute Name="email">
      <saml:AttributeValue>kim.cheolsu@jtbc.co.kr</saml:AttributeValue>
    </saml:Attribute>
    <saml:Attribute Name="Role">
      <saml:AttributeValue>ROLE_HR_ACCESS</saml:AttributeValue>
    </saml:Attribute>
  </saml:AttributeStatement>

</saml:Assertion>
```

## 4. SAML vs OIDC 비교

| 항목 | SAML 2.0 | OIDC |
|------|---------|------|
| 데이터 형식 | XML | JSON (JWT) |
| 토큰 이름 | Assertion | Access Token / ID Token |
| 사용자 정보 | AttributeStatement | Claim |
| 발급자 | Issuer | `iss` |
| 만료 | NotOnOrAfter | `exp` |
| 서명 | XML Signature (X.509) | RS256 (JWKS) |
| 전송 방식 | HTTP-POST (form) | HTTP Redirect + Bearer |
| 복잡도 | 높음 | 낮음 |
| 사용 케이스 | 레거시 기업 시스템 | 신규 개발 |

## 5. Attribute Mapper 설정

레거시 그룹웨어가 특정 속성을 요구하는 경우 Mapper 추가:

```bash
# email Attribute Mapper
curl -X POST http://keycloak/admin/realms/jtbc/clients/$CLIENT_UUID/protocol-mappers/models \
  -d '{
    "name": "email",
    "protocol": "saml",
    "protocolMapper": "saml-user-property-mapper",
    "config": {
      "user.attribute": "email",
      "attribute.name": "email"
    }
  }'

# Role Attribute Mapper
curl -X POST .../protocol-mappers/models \
  -d '{
    "name": "roles",
    "protocol": "saml",
    "protocolMapper": "saml-role-list-mapper",
    "config": {
      "attribute.name": "Role"
    }
  }'
```

## 6. Binding 방식

| Binding | 동작 | 사용 케이스 |
|---------|------|------------|
| HTTP-Redirect | SAMLRequest를 URL 파라미터로 전달 | 요청(Request) |
| HTTP-POST | SAMLResponse를 HTML form으로 전달 | 응답(Assertion) |
| SOAP | 직접 서버-서버 통신 | Artifact 해석 |

실무에서 가장 많이 쓰는 조합: **요청은 HTTP-Redirect, 응답은 HTTP-POST**

## 7. 핵심 질문 & 답

**Q. SAML Assertion과 JWT의 가장 큰 구조적 차이는?**

> JWT는 Base64url 인코딩된 JSON 3개를 점(.)으로 연결한 문자열.
> SAML Assertion은 XML 문서 전체를 XML Signature로 서명.
> JWT는 Payload 전체를 해시해서 서명하지만,
> SAML은 XML 내 특정 요소만 선택적으로 서명할 수 있음 (XML Signature의 특성).
> 이 때문에 SAML은 XML Signature Wrapping 공격에 취약할 수 있어
> 구현 시 Assertion 검증 로직을 엄격하게 작성해야 함.
