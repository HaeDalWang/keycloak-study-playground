# IAM/인증 용어집

> 헷갈릴 때마다 꺼내보는 레퍼런스

## 프로토콜 / 표준

### OAuth2
**"앱이 사용자 비밀번호를 직접 받지 않고 인증을 위임하는 프로토콜"**

- 비유: 호텔 카드키. 프런트(Keycloak)에서 카드키(Token)를 받아서 레스토랑(앱)에 제출
- 핵심: 비밀번호는 Keycloak만 알고, 앱은 Token만 받음
- 주의: OAuth2 자체는 "접근 권한 위임"이지 "신원 확인"이 아님

### OIDC (OpenID Connect)
**"OAuth2 위에 신원 정보(누구인지)를 추가한 확장 표준"**

- OAuth2 + "이 사람은 김철수입니다" = OIDC
- 추가되는 것: ID Token (사용자 정보 담긴 JWT)
- 실무에서 Keycloak 연동 = 거의 항상 OIDC 사용

```
OAuth2  →  "이 사람 접근 허용" (권한)
OIDC    →  "이 사람은 김철수, 팀장" (신원 + 권한)
```

### SAML 2.0 (Security Assertion Markup Language)
**"XML 기반의 구형 SSO 표준. 레거시 기업 시스템에서 많이 사용"**

- OIDC보다 오래됨 (2005년). 복잡하고 무거움
- 아직도 많이 씀: SAP, Oracle, 정부 시스템, 오래된 사내 포털
- Keycloak은 OIDC와 SAML 2.0 둘 다 지원
- OIDC vs SAML: 신규 개발은 OIDC, 레거시 연동은 SAML

### JWT (JSON Web Token)
**"서명된 JSON 데이터를 Base64로 인코딩한 토큰 형식"**

- 구조: `Header.Payload.Signature` (점 2개로 구분)
- Payload는 누구나 읽을 수 있음 (암호화 아님, 인코딩)
- 위변조 방지: Signature (개인키로 서명, 공개키로 검증)
- 종류: Access Token, Refresh Token, ID Token

## 역할 / 주체

### IAM (Identity and Access Management)
**"누가, 무엇에, 어떻게 접근할 수 있는지 관리하는 체계 전체"**

- Identity: 누구인가 (인증, Authentication)
- Access: 무엇을 할 수 있는가 (인가, Authorization)
- AWS IAM, Keycloak, Active Directory 모두 IAM 시스템

### IdP (Identity Provider)
**"신원을 확인하고 토큰을 발급하는 서버"**

- Keycloak = IdP
- 다른 예: Google, Kakao, 네이버 (소셜 로그인 제공자도 IdP)
- 반대 개념: SP (Service Provider) = 토큰을 받아서 쓰는 앱

### SP (Service Provider)
**"IdP에서 발급한 토큰을 받아서 서비스를 제공하는 앱"**

- HR 시스템, 사내 포털, 결재 시스템 = SP
- SP는 사용자 비밀번호를 모름. 토큰만 검증

### SSO (Single Sign-On)
**"한 번 로그인으로 여러 앱에 접근하는 것"**

- 구현 방법: OIDC 또는 SAML
- Keycloak이 SSO 세션을 관리
- 로그아웃도 SSO: 한 앱에서 로그아웃 → 모든 앱 로그아웃 (Single Logout)

## 인증 흐름

### Authorization Code Flow
**"가장 안전한 OIDC 로그인 흐름. 서버가 있는 앱에서 사용"**

```
1. 앱 → Keycloak: "로그인 페이지 주세요" (code 요청)
2. 사용자 → Keycloak: 로그인
3. Keycloak → 앱: code 발급 (브라우저 통해)
4. 앱 서버 → Keycloak: code + client_secret → token 교환
5. Keycloak → 앱 서버: Access Token 발급
```

- 핵심: client_secret이 브라우저에 노출 안 됨 (4번이 서버-서버 통신)
- 사용: 백엔드 서버가 있는 웹앱 (우리가 만든 Python 앱)

### Client Credentials Flow
**"사용자 없이 서버-서버 통신에서 사용"**

```
앱 서버 → Keycloak: client_id + client_secret → token
(사용자 로그인 없음)
```

- 사용: 마이크로서비스 간 API 호출, 배치 작업, 데몬

### PKCE (Proof Key for Code Exchange)
**"client_secret 없이 Authorization Code Flow를 안전하게 쓰는 방법"**

- 사용: 모바일 앱, SPA (client_secret을 앱에 넣으면 탈취 위험)
- code_verifier(랜덤값) → code_challenge(해시) 로 검증

## 토큰

### Access Token
**"리소스 서버(앱)에 제출하는 토큰. 짧은 유효기간"**

- 유효기간: 보통 5분 (운영 환경)
- 제출 대상: HR 시스템, 결재 시스템 등 리소스 서버
- JWT 형식: `realm_access.roles`에 권한 목록 포함

### Refresh Token
**"Access Token 만료 시 새 Access Token을 받는 토큰"**

- 유효기간: 보통 30분~수 시간
- 제출 대상: Keycloak만 (리소스 서버에 절대 보내면 안 됨)
- 탈취 시 위험: Access Token을 계속 재발급받을 수 있음

### ID Token
**"로그인한 사람이 누구인지 담긴 토큰 (OIDC 전용)"**

- 앱 자체에서 사용자 정보 표시용
- `scope=openid` 요청 시에만 발급
- 리소스 서버에 제출하는 용도가 아님

## 권한 관리

### RBAC (Role-Based Access Control)
**"역할(Role)에 권한을 부여하고, 사람에게 역할을 할당하는 방식"**

- Rule: Role = 권한, Group = 소속
- 예) ROLE_HR_ACCESS → 개발팀 그룹 → 김철수 (상속)
- 반대 개념: ABAC (속성 기반), ReBAC (관계 기반)

### Claim
**"JWT Payload 안의 각 데이터 항목"**

| Claim | 의미 |
|-------|------|
| `iss` | 발급자 (Keycloak URL) |
| `sub` | 사용자 UUID |
| `exp` | 만료 시각 (Unix timestamp) |
| `iat` | 발급 시각 |
| `azp` | 토큰을 요청한 Client |
| `realm_access.roles` | Realm Role 목록 |
| `resource_access` | Client Role 목록 |

### Scope
**"토큰에 어떤 정보를 담을지 요청하는 범위"**

- `openid`: ID Token 발급 + 기본 사용자 정보
- `profile`: 이름, 생년월일 등
- `email`: 이메일 주소
- `offline_access`: Refresh Token 장기 유지

## Keycloak 논리 단위

| 단위 | 의미 | AWS 비유 |
|------|------|---------|
| Realm | 격리된 인증 도메인 | VPC |
| Client | 보호 대상 앱 | Security Group |
| Realm Role | Realm 전체 권한 | IAM Policy (전역) |
| Client Role | 특정 앱 전용 권한 | IAM Policy (서비스 한정) |
| Group | 사용자 묶음 | IAM Group |
| User | 실제 사람 | IAM User |

## 자주 혼동하는 것들

### 인증 vs 인가
```
인증 (Authentication, AuthN): "당신이 누구인지 확인" → 로그인
인가 (Authorization, AuthZ):  "당신이 무엇을 할 수 있는지 확인" → 권한 체크
```

### Realm Role vs Client Role
```
Realm Role:  어떤 Client로 로그인해도 토큰에 포함
             → realm_access.roles
Client Role: 해당 Client로 로그인할 때만 토큰에 포함
             → resource_access.{client-id}.roles
```

### Access Token vs ID Token
```
Access Token: 리소스 서버에 제출 ("나 이 API 써도 돼")
ID Token:     앱 내부에서 사용자 정보 표시용 ("나는 김철수야")
```

### Authorization Code vs Client Credentials
```
Authorization Code: 사용자가 직접 로그인하는 경우
Client Credentials: 서버-서버 통신 (사용자 없음)
```
