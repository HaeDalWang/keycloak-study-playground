# 챕터 3: JWT 완전 분석

> 가상 시나리오: 보안팀 문의 — "토큰을 복사해서 다른 사람이 쓰면? 내용을 바꿔서 권한을 높이면?"

---

## 1. JWT 3구조

```
Header.Payload.Signature
```

| 부분 | 내용 | 특징 |
|------|------|------|
| Header | alg, typ, kid | Base64url 인코딩 (누구나 읽기 가능) |
| Payload | Claim들 (iss, sub, exp, roles...) | Base64url 인코딩 (누구나 읽기 가능) |
| Signature | RSA 서명값 | 바이너리. 위변조 방지 핵심 |

**중요**: Header와 Payload는 암호화가 아닌 인코딩. 누구나 읽을 수 있다.
보안은 Signature가 담당한다.

---

## 2. RS256 서명 메커니즘

```
[발급 시 - Keycloak]
Signature = RSA_sign( SHA256(Header.Payload), 개인키 )

[검증 시 - HR 시스템]
hash1 = SHA256(Header.Payload)          ← 받은 토큰으로 직접 계산
hash2 = RSA_decrypt(Signature, 공개키)  ← 서명 복호화
hash1 == hash2 → 유효 / 불일치 → 위변조
```

- **개인키**: Keycloak 서버 내부에만 존재. 절대 외부 노출 없음.
- **공개키**: JWKS 엔드포인트에서 누구나 조회 가능.

### JWKS 엔드포인트

```
GET /realms/jtbc/protocol/openid-connect/certs
```

- `kid` (Key ID): 토큰 Header의 `kid`와 매칭하여 검증에 사용할 공개키 선택
- 키가 여러 개인 이유: **Key Rotation** — 운영 중 키 교체 시 이전 키로 발급된 토큰이 만료될 때까지 두 키를 동시 유지

---

## 3. 위변조 시도 결과

```python
# Payload의 ROLE_HR_ACCESS → ROLE_ADMIN 으로 변조
# 서명은 원본 그대로 재사용

결과: active: False  ← 즉시 탐지
```

**이유**: Payload가 1바이트라도 바뀌면 SHA-256 해시값이 완전히 달라진다.
원본 서명은 원본 Payload 기준으로 만들어졌으므로 검증 불일치 → 거부.

## 4. 핵심 Claim 정리

| Claim | 의미 | 활용 |
|-------|------|------|
| `iss` | 발급자 URL | 앱이 "우리 Keycloak이 발급한 토큰"인지 검증 |
| `sub` | 사용자 UUID | 토큰 주인의 고유 식별자 (username은 변경 가능, UUID는 불변) |
| `azp` | 토큰을 요청한 Client | "hr-system이 요청한 토큰" |
| `exp` | 만료 시각 (Unix timestamp) | 이 시각 이후 토큰 거부 |
| `iat` | 발급 시각 | exp - iat = 유효기간 (기본 300초) |
| `realm_access.roles` | Realm 레벨 Role 목록 | RBAC 접근 제어에 사용 |
| `kid` | 서명에 사용된 키 ID | JWKS에서 공개키 조회 시 사용 |

## 5. 3종류의 토큰

| 토큰 | 제출 대상 | 유효기간 | 목적 |
|------|---------|---------|------|
| Access Token | HR 시스템 등 리소스 서버 | 짧음 (5분) | API 접근 권한 증명 |
| Refresh Token | Keycloak만 | 김 (30분~) | Access Token 만료 시 재발급 |
| ID Token | 앱 자체 (사용자 정보 표시) | Access Token과 동일 | 로그인한 사람이 누구인지 |

**Refresh Token을 리소스 서버에 보내면 안 되는 이유**:
Refresh Token은 Keycloak에서만 의미가 있다. 탈취 시 Access Token을 계속 재발급받을 수 있어 더 위험하다.

---

## 6. Access Token 유효기간과 보안

### JWT는 Stateless — 발급 후 취소 불가

```
[24시간 토큰 탈취 시나리오]
  9:00 - 토큰 발급
  9:05 - 탈취 인지 → Keycloak 세션 강제 종료
  BUT: 이미 발급된 Access Token은 여전히 유효
  → 다음날 9:00까지 HR 시스템 접근 가능 (Keycloak이 막을 방법 없음)

[5분 토큰 탈취 시나리오]
  9:00 - 토큰 발급
  9:05 - 탈취 인지 → Keycloak 세션 강제 종료
  9:10 - Access Token 자동 만료
  → 최대 5분의 피해로 제한
```

### Refresh Token이 재로그인 문제를 해결

```
9:00 - 로그인 → Access Token(5분) + Refresh Token(30분) 발급
9:05 - Access Token 만료
9:05 - 앱이 자동으로 Refresh Token → Keycloak 제출
9:05 - 새 Access Token(5분) 자동 발급
→ 사용자는 재로그인 없음. 탈취 피해 창은 5분으로 제한.
```

---

## 7. 핵심 질문 & 답

**Q. Access Token 유효기간을 24시간으로 늘리면 안 되는 이유는?**

> JWT는 Stateless라 발급 후 취소가 불가능하다.
> 탈취 시 Keycloak에서 세션을 종료해도 이미 발급된 토큰은 24시간 동안 유효하다.
> 5분 토큰은 Refresh Token으로 자동 갱신되므로 사용자 불편 없이 피해 창을 최소화한다.
