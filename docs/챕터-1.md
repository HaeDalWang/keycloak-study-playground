# 챕터 1: 기초 개념 및 아키텍처

> 가상 시나리오: JTBC 사내 포털, HR 시스템, 결재 시스템을 한 번의 로그인으로 접근하고,
> 팀장만 결재 시스템에 접근 가능하게 해달라는 고객사 요청

---

## 1. IAM (Identity and Access Management)

**"누가(Identity), 무엇에(Resource), 어떻게 접근할 수 있는가(Access)를 관리하는 체계"**

이미 AWS에서 쓰고 있다. AWS IAM에서 User를 만들고 Policy를 붙여서 S3는 접근 가능하고
EC2는 못 들어가게 하는 것이 IAM이다.

Keycloak은 그 개념을 **애플리케이션 레벨**로 가져온 것이다.

```
AWS IAM          →  Keycloak
─────────────────────────────
IAM User         →  User
IAM Policy       →  Role
IAM Group        →  Group
AWS Account      →  Realm
AWS Service      →  Client (보호 대상 앱)
```

---

## 2. RBAC (Role-Based Access Control)

**사람에게 직접 권한을 주는 게 아니라, 역할(Role)에 권한을 부여하고 사람에게 역할을 할당한다.**

```
[나쁜 방식] 직접 권한 부여
  김철수 → 결재시스템 접근 허용
  이영희 → 결재시스템 접근 허용
  ... (직원 300명이면 300번 설정, 퇴사 시 누락 위험)

[RBAC] 역할 기반
  ROLE_MANAGER → 결재시스템 접근 허용
  김철수 → ROLE_MANAGER 할당
  이영희 → ROLE_MANAGER 할당
  ... (역할만 바꾸면 끝, 퇴사 시 역할 회수 1번으로 완료)
```

## 3. OAuth2

**"애플리케이션이 사용자의 비밀번호를 직접 받지 않고, 인증을 Keycloak에 위임하는 표준 프로토콜"**

### 왜 필요한가

```
[OAuth2 없이]
  사내 포털:   "비밀번호 주세요" → 직접 저장
  HR 시스템:   "비밀번호 주세요" → 직접 저장
  결재 시스템: "비밀번호 주세요" → 직접 저장
  → 비밀번호가 3군데 저장됨. 한 곳 털리면 전부 위험.

[OAuth2]
  Keycloak(호텔 프런트): 신분증 확인 후 "카드키(Token)" 발급
  사내 포털:   카드키만 확인 (비밀번호 모름)
  HR 시스템:   카드키만 확인 (비밀번호 모름)
  결재 시스템: 카드키만 확인 (비밀번호 모름)
  → 비밀번호는 Keycloak만 알고 있음
```

### 호텔 비유 전체 구조

```
호텔 프런트  = Keycloak     (신분 확인, 카드키 발급)
투숙객       = User         (김철수, 이영희 - 실제 사람)
레스토랑     = Client       (카드키를 받아서 입장 허용하는 시설)
수영장       = Client       (마찬가지)
카드키       = Token        (로그인 후 발급받는 것)
```

## 4. OIDC (OpenID Connect)

**OAuth2는 "접근 권한 위임" 프로토콜이다. 여기에 신원 정보(Identity)까지 표준화한 것이 OIDC다.**

```
OAuth2  →  "이 사람이 접근해도 됩니다" (권한)
OIDC    →  "이 사람은 김철수이고, 팀장입니다" (신원 + 권한)
```

실무에서 Keycloak을 쓸 때는 거의 항상 OIDC를 사용한다.
OAuth2만 단독으로 쓰는 경우는 드물다.

## 5. Keycloak 논리 단위

### Realm

**완전히 격리된 인증 도메인. AWS VPC와 같다.**

- Realm 간에는 User, Client, Role이 공유되지 않는다.
- 보통 회사 하나 = Realm 하나.
- `master` Realm은 Keycloak 자체 관리용. 여기에 서비스 설정을 넣으면 안 된다.

```
master Realm  →  Keycloak 관리자 전용 (건드리지 말 것)
jtbc Realm    →  JTBC 서비스 전용 (우리가 쓰는 곳)
```

### Client ← 가장 헷갈리는 개념

**Client = 사람이 아니다. Keycloak에 등록된 애플리케이션이다.**

일상 언어에서 "클라이언트"는 고객(사람)을 뜻하지만,
OAuth2/Keycloak에서는 완전히 다른 의미다.

```
일상 언어:  클라이언트 = 고객 (사람)
OAuth2:     Client     = 애플리케이션 (HR 시스템, 사내 포털 등)
```

**Client의 역할**: Keycloak이 발급한 Token을 받아서 "이 사람 접근 허용/거부"를 판단하는 앱.
사용자의 비밀번호는 모른다. Token만 본다.

```
[인증 흐름]
  1. 김철수가 HR 시스템(Client)에 접근 시도
  2. HR 시스템이 김철수를 Keycloak으로 리다이렉트
  3. 김철수가 Keycloak에서 로그인 (비밀번호는 Keycloak만 봄)
  4. Keycloak이 Token 발급
  5. HR 시스템이 Token을 받아서 접근 허용
```

**Client authentication ON vs OFF**

| 설정 | 의미 | 사용 케이스 |
|------|------|------------|
| ON (Confidential) | client_secret 필요. 백엔드 서버만 토큰 발급 가능 | HR 시스템처럼 서버가 있는 앱 |
| OFF (Public) | client_secret 없음. 브라우저에서 직접 발급 가능 | 모바일 앱, SPA |

### Role

**권한의 묶음. RBAC의 핵심 단위.**

- Role = 권한을 표현한다. (ROLE_HR_ACCESS, ROLE_MANAGER)
- Group = 소속을 표현한다. (개발팀, 마케팅팀)
- **Role 이름은 팀 이름이 아니라 권한 이름으로 짓는다.**

```
잘못된 설계:  ROLE_DEV_EMPLOYEE  (팀 정체성 → 팀 수만큼 Role 증가)
올바른 설계:  ROLE_HR_ACCESS     (권한 → 여러 팀이 같은 Role 재사용 가능)
```

**Inherited Role**

| Inherited | 의미 | 제거 방법 |
|-----------|------|---------|
| True | Group 또는 Composite Role에서 상속 | Group에서 빼거나 Group의 Role 해제 |
| False | User에게 직접 할당 | Unassign으로 직접 제거 |

### Group

**사용자 묶음. 조직도의 부서에 해당.**

- Group에 Role을 붙이면 소속 User 전원에게 Role이 자동 상속된다.
- 인사이동 시 Group만 바꾸면 권한이 자동으로 따라온다.

## 6. 전체 구조 매핑

```
기업 조직도                    Keycloak
─────────────────────────────────────────
회사 전체 (JTBC)          →   Realm (jtbc)
  ├── HR 시스템 (앱)       →   Client (hr-system)
  ├── 사내 포털 (앱)       →   Client (intranet)
  ├── 결재 시스템 (앱)     →   Client (approval-system)
  ├── 직원 (사람)          →   User (kim.cheolsu)
  ├── HR 접근 권한         →   Role (ROLE_HR_ACCESS)
  ├── 결재 접근 권한       →   Role (ROLE_APPROVAL_ACCESS)
  └── 개발팀 (부서)        →   Group (개발팀)
                                  └── ROLE_HR_ACCESS 할당
```

## 8. 핵심 질문 & 답

**Q. 개발팀 직원 50명에게 HR 시스템 접근 권한을 주는 가장 효율적인 방법은?**

> ROLE_HR_ACCESS Role을 만들고, 개발팀 Group에 할당한다.
> 50명을 개발팀 Group에 넣으면 ROLE_HR_ACCESS가 자동 상속된다.
> Role 이름은 팀 이름(ROLE_DEV)이 아니라 권한 이름(ROLE_HR_ACCESS)으로 짓는다.
> 그래야 마케팅팀도 같은 Role을 재사용할 수 있다.
