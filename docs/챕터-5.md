# 챕터 5: CLI/API 자동화

> 가상 시나리오: 신규 부서 생성 시 수동 설정 실수 잦음 → 자동화 요청

---

## 1. kcadm.sh 개요

Keycloak 공식 CLI 도구. Admin REST API를 래핑.
모든 Admin Console UI 작업을 스크립트로 대체 가능.

```bash
# 위치
/opt/keycloak/bin/kcadm.sh

# 로그인 (이후 명령에서 토큰 자동 재사용)
kcadm.sh config credentials \
  --server http://localhost:8080 \
  --realm master \
  --user admin \
  --password admin_password
```

---

## 2. 주요 명령어

```bash
# Realm 생성
kcadm.sh create realms -s realm=jtbc -s enabled=true

# Role 생성
kcadm.sh create roles -r jtbc -s name=ROLE_HR_ACCESS

# Group 생성
GROUP_ID=$(kcadm.sh create groups -r jtbc -s name=개발팀 -i)

# Group에 Role 할당
kcadm.sh add-roles -r jtbc --gname 개발팀 --rolename ROLE_HR_ACCESS

# User 생성
USER_ID=$(kcadm.sh create users -r jtbc \
  -s username=kim.cheolsu \
  -s email=kim@jtbc.co.kr \
  -s enabled=true -i)

# 비밀번호 설정
kcadm.sh set-password -r jtbc --username kim.cheolsu --new-password Test1234!

# User → Group 추가
kcadm.sh update users/$USER_ID/groups/$GROUP_ID \
  -r jtbc -s realm=jtbc -s userId=$USER_ID -s groupId=$GROUP_ID -n

# User의 Effective Role 조회
kcadm.sh get-roles -r jtbc --uusername kim.cheolsu --effective
```

## 3. Admin REST API 직접 호출

kcadm.sh가 내부적으로 호출하는 API. 어떤 언어에서도 자동화 가능.

```bash
# 토큰 발급
TOKEN=$(curl -s -X POST http://localhost:8080/realms/master/protocol/openid-connect/token \
  -d "grant_type=password&client_id=admin-cli&username=admin&password=admin_password" \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['access_token'])")

# User 목록 조회
curl -H "Authorization: Bearer $TOKEN" \
  http://localhost:8080/admin/realms/jtbc/users

# User의 Effective Roles 조회
curl -H "Authorization: Bearer $TOKEN" \
  http://localhost:8080/admin/realms/jtbc/users/$USER_ID/role-mappings/realm/composite
```

## 4. 멱등성 (Idempotency)

**멱등성**: 몇 번을 실행해도 결과가 동일한 특성.

| 도구 | 멱등성 | 방법 |
|------|--------|------|
| Terraform | 있음 | tfstate로 현재 상태 추적 → diff만 적용 |
| kcadm.sh 스크립트 | 없음 (기본) | 직접 구현 필요 |

**스크립트에 멱등성 부여하는 방법**

```bash
# 방법 1: 존재 여부 확인 후 분기
EXISTING=$(kcadm.sh get realms/$REALM 2>/dev/null)
if [ -z "$EXISTING" ]; then
  kcadm.sh create realms -s realm=$REALM
else
  echo "이미 존재 → 스킵"
fi

# 방법 2: Realm Export/Import (Keycloak as Code)
# Realm 전체 설정을 JSON으로 export → Git 관리 → import로 적용
# 변경 이력 추적, 롤백, 환경 간 복제 가능
```

**두 번 실행 시 발생하는 문제**
- Conflict 오류로 설정 불가
- 또는 의도치 않은 중복 데이터 삽입
- Terraform의 tfstate 같은 상태 추적 메커니즘이 없으므로 더욱 신중 필요

## 5. 2노드 vs 3노드 HA (보충)

| 구성 | distributed-cache 효과 | Split-Brain 보호 | 권장 |
|------|----------------------|----------------|------|
| 2노드 + owners=2 | 없음 (= replicated) | 없음 | 최소 운영 |
| 3노드 + owners=2 | 메모리 33% 절약 | Quorum=2 보호 | 운영 권장 |
| 4노드+ | 메모리 50% 절약 | 강력 | 대규모 운영 |

2노드는 "1대 장애 시 서비스 유지"는 충족하지만,
메모리 효율과 Split-Brain 보호는 3노드부터 의미가 생긴다.

## 6. 핵심 질문 & 답

**Q. onboard.sh를 같은 Realm에 두 번 실행하면?**

> Conflict 오류 또는 중복 데이터 삽입 발생.
> Terraform은 tfstate로 현재 상태를 추적해 멱등성을 보장하지만,
> 단순 쉘 스크립트는 상태 추적이 없으므로 존재 여부 확인 로직을 직접 구현하거나
> Realm Export/Import 방식(Keycloak as Code)으로 관리해야 한다.
