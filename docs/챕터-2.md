# 챕터 2: keycloak.conf 완전 분석

> 가상 시나리오: 개발팀 테스트 완료 후 운영 서버 배포 요청

---

## 1. start-dev vs start

| 항목 | start-dev | start (운영) |
|------|-----------|-------------|
| HTTP | 허용 | 기본 차단 (HTTPS 필수) |
| DB 기본값 | H2 인메모리 | 설정 필수 |
| 빌드 최적화 | 없음 | kc.sh build 필요 |
| 테마 캐시 | 비활성 | 활성 |
| 기동 시간 | ~6초 | ~3초 (최적화) |
| 프로파일 | dev | prod |

**운영 기동 순서 (2단계 필수)**

```bash
kc.sh build   # 설정을 Quarkus 바이너리에 굽기 (bake)
kc.sh start   # 최적화된 바이너리로 기동
```

`kc.sh build`가 필요한 이유: Quarkus는 런타임이 아닌 빌드 시점에 설정을 코드에 컴파일한다.
설정이 바뀔 때마다 build → start 순서를 반복해야 한다.

## 2. keycloak.conf 핵심 파라미터

### 데이터베이스

```properties
db=postgres
db-url=jdbc:postgresql://postgres:5432/keycloak
db-username=keycloak
db-password=keycloak_password
```

**주의**: `db-url`에는 `container_name`이 아닌 Docker Compose **서비스 이름**을 사용.
폐쇄망 실제 환경에서는 DNS 장애 방지를 위해 IP 직접 사용 권장.

### 커넥션 풀

```properties
db-pool-min-size=5    # 항상 유지할 최소 연결 수
db-pool-max-size=20   # 최대 연결 수
```

**계산 공식**:
```
db-pool-max-size = (PostgreSQL max_connections × 0.8) ÷ Keycloak 노드 수

예) max_connections=100, 노드 3대
    (100 × 0.8) ÷ 3 = 26
```

**왜 80%인가**: PostgreSQL은 슈퍼유저용으로 기본 3개 예약 (`superuser_reserved_connections=3`).
나머지 20%는 모니터링 툴, 관리자 접속, 연결 급증 여유분.

**노드 증설 시 반드시 재계산 필요**: 5노드 × 20 = 100 → 한계 초과 위험.

### HTTP / 프록시

```properties
http-enabled=true
http-port=8080
proxy=edge
```

**proxy 옵션 비교**:

| 값 | 구조 | 사용 케이스 |
|----|------|------------|
| edge | [클라이언트]→HTTPS→[LB]→HTTP→[Keycloak] | 폐쇄망 Nginx/HAProxy 앞단 (가장 일반적) |
| reencrypt | [클라이언트]→HTTPS→[LB]→HTTPS→[Keycloak] | 내부 구간도 암호화 필요 시 |
| passthrough | [클라이언트]→HTTPS→[LB]→HTTPS(원본)→[Keycloak] | LB가 SSL 미처리 |
| none | 프록시 없음, Keycloak이 직접 처리 | 개발/테스트 |

### 호스트명

```properties
hostname=auth.jtbc.co.kr   # 토큰 issuer URL에 사용
hostname-strict=true        # 운영: true / 로컬 테스트: false
hostname-strict-backchannel=false
```

**hostname이 중요한 이유**: 토큰의 `iss` 클레임 값이 된다.
앱이 토큰 검증 시 `iss`를 확인하므로, hostname이 바뀌면 기존 토큰이 모두 무효화된다.

### 캐시

```properties
cache=local      # 단일 노드
# cache=ispn     # HA 클러스터 (Phase 4에서 변경)
# cache-stack=jdbc-ping  # 폐쇄망 노드 디스커버리 (Phase 4)
```

### 로깅

```properties
log-level=INFO
# 특정 패키지만 DEBUG: log-level=INFO,org.keycloak.authentication:DEBUG
log=console
```

### JVM (환경변수로 전달)

```bash
JAVA_OPTS_APPEND="-Xms4g -Xmx4g"
```

**메모리 공식**:
- 서버 RAM의 50~70%를 Keycloak에 할당
- `Xms = Xmx` 동일하게 설정 (힙 크기 변동으로 인한 GC 부하 방지)
- 예) 서버 RAM 8GB → `-Xms4g -Xmx4g`

## 3. 실습 중 발생한 장애 2건

### 장애 1: `Unknown option: 'sh'`

```
원인: Keycloak 이미지 ENTRYPOINT = /opt/keycloak/bin/kc.sh (고정)
      docker-compose command에 넣은 값이 kc.sh의 인자로 전달됨
      → kc.sh sh -c "..." 가 되어버림

해결: command 대신 entrypoint를 오버라이드
      entrypoint: ["/bin/bash", "-c", "kc.sh build && kc.sh start"]
```

### 장애 2: `Failed to obtain JDBC connection / keycloak-db`

```
원인: keycloak.conf의 db-url에 container_name(keycloak-db)을 사용
      Docker 네트워크에서 컨테이너 간 통신은 서비스 이름으로 해야 함

해결: db-url = jdbc:postgresql://postgres:5432/keycloak
      (docker-compose.yml services: 키 이름 사용)
```

## 4. 핵심 질문 & 답

**Q. PostgreSQL max_connections=100, 노드 3대 → db-pool-max-size=20 유지 시 문제는?**

> 3 × 20 = 60으로 100 미만이므로 수치상 문제없음.
> 단, DB를 다른 앱과 공유하거나 노드 증설 계획이 있다면 위험.
> 안전 공식: (100 × 0.8) ÷ 3 = 26으로 설정하고,
> 노드 증설 시마다 재계산 필수.
