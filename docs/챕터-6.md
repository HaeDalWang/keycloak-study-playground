> 가상 시나리오: 장애 유형별 탐지/대응/복구 프로세스 + 26.x LTS 전환

## 1. 장애 시나리오별 대응 프로세스

### 시나리오 1: 노드 다운 (Phase 4에서 실습 완료)

```
탐지: nginx 헬스체크 실패 → 자동 제외
대응: 자동 (nginx가 살아있는 노드로 라우팅)
복구: docker start <노드> → 자동 클러스터 재조인 + 세션 리밸런싱
```

### 시나리오 2: DB 연결 차단

```
[핵심 교훈] DB 차단 직후에는 커넥션 풀이 살아있어 일시적 정상 동작
→ 모니터링에서 "정상"으로 보임 (좀비 상태)
→ 풀 고갈 후 갑자기 전면 장애 → 초기 대응 지연

탐지: 로그인 성공률 모니터링만으로는 부족
      → DB 커넥션 풀 지표 직접 모니터링 필요
      → /metrics 엔드포인트의 agroal_pool_active_count

대응: 방화벽 규칙 확인 및 복구
복구: 네트워크 복구 후 커넥션 풀 자동 재연결
```

### 시나리오 3: 키 로테이션

```
[올바른 절차]
1. 새 키(kid=B) 추가 → 이후 발급 토큰은 kid=B 사용
2. 기존 키(kid=A)는 DEPRECATED 유지 (검증 가능, 발급 불가)
3. 기존 토큰 exp 이후 → kid=A 삭제

[잘못된 절차 - 절대 금지]
기존 키 즉시 삭제 → 모든 로그인 사용자 강제 로그아웃
```

### 장애 대응 총정리

| 장애 유형 | 탐지 방법 | 즉각 대응 | 복구 |
|---------|---------|------|------|
| 노드 다운 | /health/ready 실패 | nginx 자동 제외 | 노드 재기동 |
| DB 연결 차단 | 커넥션 풀 지표 + 로그인 실패 | 방화벽 확인 | 네트워크 복구 |
| DB 연결 지연 | 응답시간 증가, 타임아웃 로그 | DB 부하 확인 | 풀 크기 조정 |
| 키 유출 | JWT 검증 실패 로그 | 즉시 키 로테이션 | 올바른 절차로 교체 |
| Split-Brain | 클러스터 뷰 불일치 | 소수 노드 격리 | 네트워크 복구 후 재조인 |

## 2. Keycloak 24.x → 26.x 업그레이드

### 버전 현황 (2026년 5월 기준)

- **최신 버전**: 26.6.1 (2026년 4월 15일)
- **24.x 상태**: EOL 수순 (비권장)
- **26.x**: 현재 지원 버전 (Full Support)

### 주요 변경사항

| 항목 | 24.x | 26.x |
|------|------|------|
| Quarkus | 3.8.3 | 3.33.1 |
| Organization | 없음 | 추가 (멀티테넌시) |
| Token Exchange | 실험적 | 정식 지원 |
| `password` grant | 지원 | Deprecated 경고 |
| Admin Console | 구버전 혼재 | 신규 UI 완전 전환 |

### keycloak.conf 변경사항

```properties
# 24.x → 26.x 호환되는 설정 (하위 호환 유지)
# 단, hostname-v2 관련 옵션 추가됨
hostname-admin=http://localhost:8080  # 신규: Admin Console URL 별도 지정
```

### Organization 기능 활성화

```bash
# Realm에 Organization 기능 활성화
kcadm.sh update realms/jtbc -s "organizationsEnabled=true"

# Organization 생성
kcadm.sh create organizations -r jtbc \
  -s name="JTBC 드라마팀" \
  -s domains='[{"name":"drama.jtbc.co.kr"}]'
```

### `password` grant Deprecated 대응

```bash
# 26.x에서 deprecated (실습 편의상 사용했지만 운영 금지)
grant_type=password

# 운영에서는 authorization_code flow 사용
# 앱이 Keycloak 로그인 페이지로 리다이렉트 → 코드 교환 → 토큰 발급
```

### 업그레이드 방법

```yaml
# docker-compose.yml 이미지 태그만 변경
image: quay.io/keycloak/keycloak:26.6.1  # 24.0.3 → 26.6.1
```

DB 스키마는 Keycloak이 자동 마이그레이션 (Liquibase).
설정 파일(keycloak.conf) 구조는 거의 동일.

## 3. 핵심 질문 & 답

**Q. DB 연결이 차단됐는데 처음에는 로그인이 됐습니다. 왜 그랬습니까?**

> 커넥션 풀(db-pool-min-size=5)에 이미 맺어진 연결이 살아있어서
> 차단 직후에는 기존 풀 연결로 요청을 처리했기 때문이다.
> 이 현상이 위험한 이유는 모니터링에서 "정상"으로 보이는 동안
> 실제로는 장애가 확정된 상태이기 때문이다.
> 풀이 고갈되면 갑자기 전면 장애가 발생하고 초기 대응이 늦어진다.
> 로그인 성공률이 아닌 DB 커넥션 풀 지표를 직접 모니터링해야 한다.
