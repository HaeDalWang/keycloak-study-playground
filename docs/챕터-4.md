# 챕터 4: HA 클러스터링

> 가상 시나리오: 단일 노드 Keycloak 장애 시 전사 서비스 중단 → 이중화 요청

---

## 1. SPOF 문제와 HA 목표

```
[단일 노드 SPOF]
  직원 → [Keycloak 1대] → 장애 → 전사 서비스 중단

[HA 목표]
  직원 → [Nginx LB] → [Keycloak-1]
                    → [Keycloak-2]
  노드1 다운 → 노드2가 자동 처리
```

단순히 노드를 2대로 늘리면 세션이 한 노드 메모리에만 존재해서
다른 노드로 라우팅 시 "세션 없음" 오류 발생.
→ Infinispan 분산 캐시로 해결.

---

## 2. Infinispan 캐시 타입

| 타입 | 동작 | 메모리 | 사용 케이스 |
|------|------|--------|------------|
| `replicated-cache` | 모든 노드에 전체 복사 | 높음 | 인증 진행 중 세션 (authenticationSessions) |
| `distributed-cache` | 기본 2개 노드에 분산 저장 | 낮음 | 로그인 완료 후 세션 (sessions) |

**왜 두 가지를 구분하는가**

- `authenticationSessions` (replicated): 로그인 진행 중 사용자가 여러 노드를 오갈 수 있음. 데이터가 작고 수명이 짧으므로 전체 복사 부담 없음. 어느 노드에서도 인증 상태 유지 필수.
- `sessions` (distributed): 로그인 완료 후 세션. 데이터가 크고 수명이 김. 모든 노드 복사 시 메모리 낭비. 기본 2개 노드에만 저장 → 1개 노드 다운돼도 나머지 1개에 데이터 유지.

---

## 3. 노드 디스커버리 방식

| 방식 | 동작 | 사용 환경 |
|------|------|---------|
| 멀티캐스트(UDP) | UDP 브로드캐스트로 노드 탐색 | 일반 LAN (폐쇄망에서 차단됨) |
| TCPPING | 노드 IP/호스트 목록 직접 지정 | IP 고정 환경, Docker Compose |
| JDBC_PING | DB 테이블을 노드 등록부로 사용 | 폐쇄망 운영 환경 (IP 유동적) |

**Keycloak 24.x 주의사항**: `cache-stack=jdbc-ping`은 직접 지원 안 됨.
JDBC_PING 사용 시 `--cache-config-file`로 커스텀 Infinispan XML 제공 필요.

## 4. 핵심 설정 변경 (단일 노드 → HA)

```properties
# Phase 2 (단일 노드)
cache=local

# Phase 4 (HA 클러스터)
cache=ispn
cache-stack=tcp
health-enabled=true   # 헬스체크 엔드포인트 활성화 (K8s, LB 필수)
```

## 5. 실습 중 발생한 장애 3건

### 장애 1: `Invalid value for option 'kc.cache-stack': jdbc-ping`

```
원인: Keycloak 24.x에서 jdbc-ping은 cache-stack 직접 값으로 미지원
      지원 값: tcp, udp, kubernetes, ec2, azure, google

해결: cache-stack=tcp + TCPPING으로 대체
     실제 운영 JDBC_PING은 커스텀 cache-ispn.xml 필요
```

### 장애 2: `duplicate key value violates unique constraint "uk_cli_scope"`

```
원인: 두 노드가 동시에 기동하며 DB 초기화 Race Condition 발생
      노드1, 노드2가 동시에 동일한 초기 데이터 삽입 시도

해결: 노드1에 healthcheck 추가
      노드2는 노드1이 healthy 상태가 된 후에 기동 (depends_on 조건)
```

### 장애 3: `curl/wget not found` (헬스체크 실패)

```
원인: Keycloak UBI 이미지에 curl, wget 미포함

해결: bash /dev/tcp 트릭으로 HTTP 요청 직접 구성
      exec 3<>/dev/tcp/localhost/8080 && echo -e 'GET /health/ready...' >&3
```

## 6. 장애 시나리오 실습 결과

```
[노드1 기동] → 클러스터 코디네이터로 단독 시작
[노드2 기동] → 노드1 클러스터에 조인
              Finished rebalance with members [노드1, 노드2]

[노드1 강제 종료]
  nginx → 노드1 (❌) → 자동으로 노드2로 재시도
  노드2의 distributed-cache에 세션 데이터 있음
  → HTTP 200 OK (서비스 중단 없음)
```

## 7. 핵심 질문 & 답

**Q. 노드1 세션이 노드2에서도 유효한 이유? replicated vs distributed 선택 기준?**

> Infinispan distributed-cache가 세션 데이터를 기본 2개 노드에 복제하므로,
> 노드1이 다운돼도 노드2에 복사본이 존재한다.
>
> 로그인 완료 후 세션(sessions)은 distributed-cache를 써야 한다.
> 세션 데이터는 크고 수명이 길어 모든 노드에 복사(replicated)하면 메모리 낭비가 심하다.
> distributed는 기본 2개 노드에만 저장해 메모리를 절약하면서도 1개 노드 장애를 허용한다.
>
> 인증 진행 중(authenticationSessions)은 replicated-cache를 쓴다.
> 로그인 플로우 중 어느 노드로 요청이 가도 인증 상태가 있어야 하고,
> 데이터가 작고 수명이 짧아 전체 복사 부담이 없다.
